# RAG 求职知识库助手 — 演示文案（4 分钟）


## 第 1 分钟：讲场景

> **一句话概括**：这个项目解决了面试场景下「信息分散、无法实时检索」的问题，目标用户是求职者本人和面试官双方。
>
> 传统简历是静态 PDF，面试官想问简历上没写全的东西只能靠猜。而 RAG 可以把求职者的项目经历、教育背景、技能特长、性格特点全部整理成结构化的知识库，面试官自然语言提问，系统检索最相关的文档，再让 LLM 生成带来源引用的回答。
>
> 为什么用 RAG 而不是微调？因为个人知识库的内容会持续更新——今天学了一个新框架、明天完成一个新项目——RAG 只要加文档重新索引就行，不需要重新训练模型。

---

## 第 2 分钟：讲架构

> 整体分三层。
>
> **用户层**是 React 前端，面试官在聊天框输入问题，通过 SSE 流式接收回答。
>
> **中间层**是 FastAPI 后端，收到问题后走三条链路：
> 第一，调 Embedding 服务（Qwen3-Embedding-8B）把问题转成 1024 维向量；
> 第二，去 MySQL 里读出所有文档块的向量，用 numpy 算余弦相似度，取 Top-5 最相关的文本块；
> 第三，把检索到的文本块拼成上下文，连同 system prompt 和历史对话一起发给 DeepSeek-V4-Flash 生成回答，再 SSE 逐字推回前端。
>
> **底层**是 Docker Compose 编排的三个容器——MySQL、FastAPI、Nginx，部署在云服务器上。Nginx 做统一反向代理，把 /api/ 转发到后端，转发 /health 做健康检查，同时记录独立访问日志。
>
> 这里有个设计取舍：LLM 和 Embedding 在管理后台可以走不同的 provider，互不依赖。比如 LLM 用 DeepSeek，Embedding 用 OpenAI 或者本地 Ollama，在后台分开填 Key 和 Base URL 就行，不需要改代码重启。

## 第 3 分钟：现场演示

### 正常场景

> 我先问一个正常问题：「林知行的 MBTI 是什么？」
>
> （等待响应）
>
> 大家可以看到，右边出来了来源卡片，显示相似度 57%，来自《个人基础信息》这个文档。下方 AI 在流式返回回答：INTJ，建筑师型，并详细列出了四个维度的特征——内倾、直觉、思维、判断。每个关键信息都能在展开的来源文档里找到原文对应，不是模型瞎编的。
>
> 这个过程实际发生了什么：问题在前端输入，后端收到后做了 Embedding 检索，取到相关文档块，拼成上下文给 LLM，LLM 生成回答后 SSE 逐字推回。整个过程在 3 到 5 秒内完成。

### 边界场景

> 再试一个文档里没有答案的问题：「肯德基全家桶多少钱？」
>
> 大家看到，系统虽然没有找到强相关的文档，但它没有胡说八道。检索到的最低分 source 低于我们设定的 0.15 阈值，引擎直接返回了空结果，前端提示「未在知识库中找到相关信息」。这就是 system prompt 里「不确定就说不知道」在起作用，也是 RAG 对比纯 LLM 的核心优势——你可以控制它不发挥。

### 失败恢复场景

> 最后演示一下输入异常。我传一个空字符串……
>
> 后端直接返回了 HTTP 400，错误信息是结构化的 JSON，前端可以捕获并给出提示，不会崩页面。这是我们统一了错误处理后的效果。

## 第 4 分钟：讲指标与取舍

> 我准备了 20 条测试问题，覆盖基本信息、教育、项目、技能、性格、职业规划六个类目。
>
> **当前指标**：
> - 检索命中率：100%（20/20）
> - 平均检索得分：0.37（满分 1.0，余弦相似度）
> - 典型响应时间：3-5 秒首 token
> - 来源引用：每个回答附带 Top-5 来源文档名和相似度评分
>
> **对比过的方案**：我测试了三种 chunk size——300、500、700 字。300 的检索平均得分最高（0.394），但因为块数多了近一倍，首 token 延迟也更长。最终选了 500 做平衡。
>
> **当前缺点**：
> 一是宽泛问题的召回质量不够好，比如「他是个什么样的人」这种问题太散，单个 chunk 匹配度不高，LLM 很难组织出有深度的回答。二是还没有系统跑过 Faithfulness 评测——虽然我手工看了几个回答都没问题，但缺少自动化机制。
>
> **下步优化**：第一加 query rewrite——宽泛问题先让 LLM 拆成 2-3 个子问题再分别检索、合并排序；第二加 rerank 层，对 Top-K 做交叉编码器重排，提升 Faithfulness；第三跑自动化评测脚本，每次发版前自动执行 20 条测试并输出报告。


## 演示注意事项

| 不要 | 要 |
|---|---|
| 只讲技术名词 | 讲完名词接一句「为什么这么选」 |
| 只点页面不解释流程 | 每一步操作都对应到架构里的一个环节 |
| 只展示完美结果 | 演示一个失败场景并分析原因 |
| 说「这个很简单」 | 说「这里我踩过坑，原因是……」 |
| 隐瞒缺点 | 主动说出当前局限和下步改进方向 |
